# ELF2 开发板 $\leftrightarrow$ 服务器通信协议约束文档 (V1.0)

## 1. 协议设计概述

### 1.1 核心设计原则

- **时序严谨性**：网络传输存在抖动与延迟，所有数据包在诞生时必须在板载端（ELF2）打上本地系统的**毫秒级 Unix 时间戳**，服务器必须以包内时间戳作为多模态数据对齐的唯一基准。
- **物理防粘包/断包**：采用 **TLV (Type-Length-Value)** 变长架构。底层套接字（Socket）接收端必须先通过固定帧头对齐，再通过 `Length` 字段精准截断数据流。
- **全局数据校验**：每帧末尾包含 4 字节的 `CRC32` 校验码，校验范围涵盖**帧头+载荷**，不满足校验的数据包必须立即丢弃，防止坏数据污染算法引擎。

### 1.2 ROS2 适配架构建议

在 ELF2 开发板中，建议设计一个专用的 **“通信桥接节点 (Communication Bridge Node)”**：

1. 该节点订阅 ROS2 中的注意力话题、小车里程计/控制话题（`geometry_msgs/msg/Twist`）及情绪话题。
2. 该节点通过底层 SPI/串口驱动或 BrainFlow 接收原始脑电。
3. 节点内部统一调用本协议打包器，通过同一个 TCP 套接字发送至服务器。

## 2. 帧结构总览 (Global Frame Structure)

无论是何种数据类型，出板的二进制流必须严格套用以下统一模具：

```
+-----------------------------------------------------------------------------------------+
|                  帧头段 (Header - 固定 15 字节)                  | 载荷段 | 校验段 |
+------------+------------+-----------+--------------+------------+--------+------------+
|  同步魔数  |  帧序列号  |  消息类型 |  Payload长度 | 毫秒时间戳 | Payload|   CRC32    |
| (2 Bytes)  | (2 Bytes)  |  (1 Byte) |  (4 Bytes)   | (8 Bytes)  | (N B)  | (4 Bytes)  |
+------------+------------+-----------+--------------+------------+--------+------------+
```

### 2.1 帧头与帧尾字段详细定义

| **字节偏移**      | **字段名**                 | **数据类型** | **字节数** | **约束与说明**                                               |
| ----------------- | -------------------------- | ------------ | ---------- | ------------------------------------------------------------ |
| `0 ~ 1`           | **Sync Header** (同步魔数) | `uint16_t`   | 2          | **固定为 `0xAA55`**。大端模式传输。用于在流式 TCP 中精确定位帧起始点。 |
| `2 ~ 3`           | **Frame ID** (帧序列号)    | `uint16_t`   | 2          | $0 \sim 65535$ 循环递增。服务器用其校验是否发生网络丢帧。    |
| `4`               | **Msg Type** (消息类型)    | `uint8_t`    | 1          | 标志当前帧的业务类型，具体取值见 **第3节**。                 |
| `5 ~ 8`           | **Length** (载荷长度)      | `uint32_t`   | 4          | **仅指 Payload 的绝对字节数 $N$**。不包含帧头和 CRC 本身的长度。 |
| `9 ~ 16`          | **Timestamp** (时间戳)     | `uint64_t`   | 8          | ELF2 封装该数据时的 **Unix 时间戳（毫秒级）**。              |
| `17 ~ (17+N-1)`   | **Payload** (数据载荷)     | `uint8_t[]`  | $N$        | 核心业务数据，长度由 `Length` 决定，解析方式由 `Msg Type` 决定。 |
| `(17+N) ~ (20+N)` | **CRC32** (校验码)         | `uint32_t`   | 4          | 对 `Sync Header` 到 `Payload` 最后一个字节的全局 CRC32 校验值。 |

*注：所有多字节字段（如 `uint32_t`, `float`）在网络传输时必须硬性规定为 **大端模式 (Big-Endian / Network Byte Order)**。*

## 3. 消息类型 (Msg Type) 载荷（Payload）详细约束

### 3.1 Type `0x01`：注意力程度 (Attention Level)

- **业务含义**：经算法解析后的用户实时注意力数值。
- **发送属性**：可周期发送，也可在数值发生明显变化时突发。
- **Payload 长度 ($N$)**：固定为 **4 字节**。

**Payload 内部字节映射：**

| **相对偏移** | **字段名**        | **数据类型** | **字节数** | **取值范围与约束**                                           |
| ------------ | ----------------- | ------------ | ---------- | ------------------------------------------------------------ |
| `0 ~ 3`      | `attention_score` | `float`      | 4          | 单精度浮点数。建议归一化控制在 $0.0 \sim 1.0$ 或 $0.0 \sim 100.0$。 |

### 3.2 Type `0x02`：小车控制与运动状态 (Robot Velocity)

- **业务含义**：小车的实时线速度与角速度。完全适配 ROS2 的 `geometry_msgs/msg/Twist` 核心数据。
- **发送属性**：周期性发送（通常与 ROS2 控制频率同步，如 20Hz 或 50Hz）。
- **Payload 长度 ($N$)**：固定为 **8 字节**。

**Payload 内部字节映射：**

| **相对偏移** | **字段名**         | **数据类型** | **字节数** | **约束与说明**                                               |
| ------------ | ------------------ | ------------ | ---------- | ------------------------------------------------------------ |
| `0 ~ 3`      | `linear_velocity`  | `float`      | 4          | 线速度 (m/s)，对应 ROS2 中 `twist.linear.x`。正数前进，负数后退。 |
| `4 ~ 7`      | `angular_velocity` | `float`      | 4          | 角速度 (rad/s)，对应 ROS2 中 `twist.angular.z`。正数左转，负数右转。 |

### 3.3 Type `0x03`：三分类情绪概率 (Emotion Probability)

- **业务含义**：同步传输积极（Positive）、中性（Neutral）、消极（Negative）三种情绪的实时百分比。
- **发送属性**：通常由脑电情绪分类模型周期性（如每秒 1~2 次）或突发异步更新。
- **Payload 长度 ($N$)**：固定为 **12 字节**。

**Payload 内部字节映射：**

| **相对偏移** | **字段名**       | **数据类型** | **字节数** | **取值范围与约束**                  |
| ------------ | ---------------- | ------------ | ---------- | ----------------------------------- |
| `0 ~ 3`      | `positive_ratio` | `float`      | 4          | 积极情绪概率，范围 $0.0 \sim 1.0$。 |
| `4 ~ 7`      | `neutral_ratio`  | `float`      | 4          | 中性情绪概率，范围 $0.0 \sim 1.0$。 |
| `8 ~ 11`     | `negative_ratio` | `float`      | 4          | 消极情绪概率，范围 $0.0 \sim 1.0$。 |

*注：严谨性约束——服务器端在解析此包时，可校验 `positive_ratio + neutral_ratio + negative_ratio == 1.0`（允许微小的浮点误差），不满足则视为无效推理结果。*

### 3.4 Type `0x04`：原始/预处理脑电波形流 (EEG Waveform)

- **业务含义**：多通道连续脑电信号的时域原始 ADC 数值或滤波后数据。支持自适应通道和变长采样打包。
- **发送属性**：高频高实时周期发送（例如每积攒 40ms 数据打包发送一次）。
- **Payload 长度 ($N$)**：动态可变。计算公式：$N = 3 + (\text{Channels} \times \text{Samples} \times 4)$ 字节。

**Payload 内部字节映射：**

| **相对偏移** | **字段名**       | **数据类型** | **字节数**                                       | **约束与说明**                                               |
| ------------ | ---------------- | ------------ | ------------------------------------------------ | ------------------------------------------------------------ |
| `0`          | `channels`       | `uint8_t`    | 1                                                | 本帧包含的脑电通道数。例如 4 通道芯片填 `4`。                |
| `1 ~ 2`      | `samples`        | `uint16_t`   | 2                                                | 每个通道在这一帧里打包的采样点数。例如打包了 10 个时间点的数据填 `10`。 |
| `3 ~ N-1`    | `eeg_data_array` | `int32_t[]`  | $\text{Channels} \times \text{Samples} \times 4$ | 脑电数据矩阵。元素为有符号 32 位整型（对应 24 位 ADC 扩充符号位）。 |

**`eeg_data_array` 内存紧凑排列顺序规范：**

必须采用“先跨通道，再跨时间点”的排列方式。假设有 2 个通道 ($C_1, C_2$)，打包了 3 个采样点 ($S_1, S_2, S_3$)，则字节流中的数据物理排列顺序必须为：

$$[C_1S_1, C_2S_1, C_1S_2, C_2S_2, C_1S_3, C_2S_3]$$

### 3.5 Type `0x00`：链路心跳包 (Heartbeat)

- **业务含义**：在没有任何数据需要传输的时间段内，网络传输线程应每隔 1~2 秒发送一次心跳包，用以维持 TCP 长连接，防止被网络路由节点或服务器防火墙强行断开。
- **Payload 长度 ($N$)**：固定为 **0 字节**（无载荷，帧头后直接跟 CRC32）。

## 4. 异常处理与边界约束

1. **时钟同步约束**：由于 ELF2 负责打毫秒时间戳，ELF2 开发板必须在联网时通过 NTP（网络时间协议）与服务器或标准时间服务器进行严格的时钟同步，避免时钟漂移导致时序对齐失败。
2. **内存对齐约束**：在任何语言（如 C/C++ 结构体或 Python Struct）实现该协议时，必须开启**1字节对齐**（C 语言使用 `#pragma pack(1)`），严禁编译器在帧内自动插入空字节。
3. **安全接收边界判断**：
   - 若服务器解析到的 `Length` 超过系统允许的最大载荷（例如设定脑电单包载荷不得超过 64KB），则判定为网络流严重损坏。
   - **处置机制**：服务器必须断开当前 socket 连接，清空缓冲区，等待 ELF2 重新连接。
